# Decision Log

**Phase ordering: render-logic before visual restyle.** Phase 1 swaps the data source (HealthSection → store slice) and rewrites the obsolete unit tests, but does not change the layout. Phase 2 then changes the visual shell on top of the now-correct data flow. Alternatives considered: (a) one combined renderer phase; (b) visual-first then data-source swap. Rejected (a) because the implementer needs a natural review checkpoint between "data-source correct, tests green" and "visuals match the spec, tests still green" — bundling them risks a single PR-sized blob with partial fixes hidden inside. Rejected (b) because the new layout assumes the bottom health-line and the unreachable-explainer are already store-driven; doing visuals first would require two passes over the same JSX. The chosen split lets each phase end on a green typecheck / lint / unit-test gate with no half-deleted modules.

**Why Phase 3 (e2e) is its own phase, not bundled into Phase 2.** The e2e suite is launched as a separate npm task and gates a different verification step. Bundling would mean Phase 2's "complete" gate has to include an e2e run (slow, flaky, headed-browser dependency on the fake-claude harness). Splitting keeps Phase 2's gate fast (vitest + tsc + eslint) and isolates the e2e fix to a single 3-line patch. The e2e suite is **broken** between the end of Phase 2 and the end of Phase 3 — but this is acceptable because the e2e suite is not run in the per-phase verifier; it is a final-check / CI gate that runs once per cycle.

**Inline recents list, no shared component.** Mirrors the decision already locked in by Phase 3 of the canvas redesign cycle (`ProjectPanel.tsx` inlines its own restyled list rather than depending on `RecentsList`). Brief Q3 confirms this. Alternative considered: keep `RecentsList.tsx`, restyle it, share between `ProjectPanel` and `LaunchGate`. Rejected because the two surfaces have different layout constraints (sidebar 16px gap vs. centred column 24px gap; sidebar excludes the current project, gate has no current project to exclude) and the shared abstraction would need props for both — a premature abstraction for two callers. Keep them parallel and inlined.

**Section-header role pattern.** Use `<div role="heading" aria-level={2}>` for `RECENT` and `OPEN` instead of `<h2>`. This matches `ProjectPanel.tsx:137,166` exactly and is what the brief specifies (locked palette tokens). The `<h1>` for "Atrium" stays as a real `<h1>` (or `<div role="heading" aria-level={1}>` — implementer's choice per brief Q4) because the document outline benefits from one true heading.
