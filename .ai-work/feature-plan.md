# Plan: Project Scaffolding (Stage 01)
_Brief: `.ai-work/interview-brief.md`_
_Created: 2026-04-17_

## Overview
Bootstrap the Atrium Electron app from zero into a running skeleton: `electron-vite` + TypeScript + React 19 + Vitest + ESLint 9 (flat) + Prettier, with the standard `src/{main,preload,renderer,shared}` layout, strict composite TS configs, path aliases wired in both tsconfig and `electron.vite.config.ts`, a smoke test proving Vitest works, and a launchable window that visually proves React mounted. No domain functionality — every subsequent stage plugs into this substrate. The plan is structured so each phase produces an independently verifiable artifact and the final phase locks the whole scaffold into one commit.

## Decision Log

### D1. Phase ordering: configs before source
**Chosen:** Write tsconfigs and `electron.vite.config.ts` before any `.ts`/`.tsx` source file.
**Alternatives:**
- Source first, then configs — source files written without type-check coverage cause cascading errors that are hard to isolate from config mistakes.
- Interleaved (write each source file with its own config slice) — fragments the config surface across phases; makes alias drift harder to catch.
**Why:** Configs are the contract. Putting them in place first means the moment source lands, `tsc -b` gives a clean pass/fail signal on the scaffold alone, not on a mix of config + source bugs.

### D2. Preload path resolution via `import.meta.dirname`
**Chosen:** In `src/main/index.ts`, compute the preload path as `path.join(import.meta.dirname, '../preload/index.js')` (relative to compiled `out/main/index.js`).
**Alternatives:**
- `__dirname` — not available in ESM without the `createRequire` dance.
- Hardcoded absolute path — breaks once packaged (Stage 06).
- `app.getAppPath()` — works but couples to Electron's runtime, which is harder to reason about in tests.
**Why:** `import.meta.dirname` is first-class in Node 24 ESM, matches `electron-vite`'s out-dir layout, and stays correct after electron-builder repackaging later.

### D3. `sandbox: false` for the preload despite the security trade-off
**Chosen:** `sandbox: false` at Stage 01 (per brief).
**Alternatives:** `sandbox: true` — would force the preload into a sandboxed context which restricts what `contextBridge` can import from Node builtins.
**Why:** Stage 02 needs the preload to expose IPC bindings that rely on Node module resolution in the preload bundle. Turning sandbox on later is a one-line reversal if we decide to harden; turning it off later would break whatever Stage 02 built. The brief made this call; documenting so future stages don't re-litigate it.

### D4. Separate `tsconfig.node.json` and `tsconfig.web.json` (no monolith)
**Chosen:** Three composite tsconfigs: root (references), node, web.
**Alternatives:**
- Single `tsconfig.json` with merged `lib` and `types` — produces incorrect globals (e.g. `window` leaking into main-process files, `process` leaking into renderer) and loses per-target `moduleResolution`.
**Why:** Per-target lib/types are the whole reason `@electron-toolkit/tsconfig` ships split bases. Merging would defeat it and invite cross-process type bugs that don't surface until Stage 02+.

### D5. Path aliases live in two places and we accept the drift risk
**Chosen:** Define aliases in tsconfigs **and** `electron.vite.config.ts`; add a mirror-warning comment at both sites pointing to `STACK_VERSIONS.md`.
**Alternatives:**
- Single-source-of-truth plugin (`vite-tsconfig-paths`) — adds a dep just to bridge two config systems that both need explicit declarations anyway for their own tooling.
- Auto-generated aliases from directory structure — brittle; conflates "what Vite bundles" with "what TS resolves."
**Why:** The duplication is four lines per site; a plugin masks it but doesn't eliminate the failure mode (a typo still breaks one side). An explicit comment makes drift a code-review smell, which is cheaper than hunting the plugin's generated output.

### D7. ESLint `projectService: true` over legacy `project: [...]`
**Chosen:** Configure `typescript-eslint` with `parserOptions.projectService: true`.
**Alternatives:**
- `parserOptions.project: ['./tsconfig.node.json', './tsconfig.web.json']` — forces ESLint to explicitly know which tsconfig owns each file; breaks on `eslint.config.js` itself (not in any include).
- Add `eslint.config.js` to `tsconfig.node.json`'s `include` — couples tsconfig to lint tooling; awkward and widens the TS project unnecessarily.
**Why:** `projectService` (typescript-eslint ≥ v8) auto-discovers the relevant tsconfig per file via TypeScript's project-service API. It handles config-file linting correctly without hacks, matches upstream's current recommendation, and avoids the "file not in project" footgun that the legacy `project` approach trips on at Stage 01's very first `npm run lint`.

### D6. Smoke test in `src/shared/__tests__/` not at repo root
**Chosen:** `src/shared/__tests__/smoke.test.ts`.
**Alternatives:** Root-level `__tests__/` — falls outside the TS project graph and needs its own tsconfig.
**Why:** Inside `src/shared/` it's covered by `tsconfig.node.json` (shared compiles in node target until a real split is needed) and by the Vitest main project. One extra file, zero extra config.

## Edge Cases in Implementation

1. **Pre-existing files (`CLAUDE.md`, `.ai-arch/`, `.claude/`, `.ai-work/`)** — Phase 1 leaves them untouched; Phase 9 step 2 explicitly stages them alongside scaffold additions into the single initial commit.
2. **`.git` already exists** — Phase 1 step 1 detects this and skips `git init`. If the working tree is dirty in a way that would contaminate the scaffold commit, it stops and surfaces the state to the user rather than force-adding.
3. **`package-lock.json` behavior** — Phase 2 step 7 accepts the lockfile as generated by `npm install`. No `npm ci`, no `prune`, no regeneration.
4. **Node version mismatch at implementation time** — Phase 1 step 6 emits a warning if `node -v` doesn't satisfy `engines.node`, but does not abort. Developer's call.
5. **Path alias drift** — Phase 3 and Phase 4 each put a mirror-warning comment at the top of their config block pointing to `STACK_VERSIONS.md`. Any change that touches one site will trip review on the other via grep for the comment.
6. **Windows path separators** — Every config file uses forward slashes in string literals. `path.resolve` / `path.join` handle platform normalization. No hardcoded `\`. Specifically called out in Phase 4 step 1 and Phase 5 step 1.
7. **Electron ESM + `sandbox: false` preload** — Phase 5 step 2 keeps the preload file minimal; Phase 8 step 6 verifies the preload actually loads via "no preload errors in DevTools Console" — this is the live-fire check the brief asks for.
8. **Smoke test only / empty renderer project** — Phase 7 step 4 has an explicit fallback: if Vitest trips on the empty renderer project at runtime, remove the project with a TODO referencing Stage 02 rather than installing `jsdom` now.
9. **DevTools detached on small displays** — Acknowledged not blocking; Phase 8 step 6 verifies detached mode works; no special handling.
